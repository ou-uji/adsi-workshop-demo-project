# 有給休暇申請機能 — 要件 Q&A（確定）

> ブランチ: `feature/skill-auto-dev`
> 参照モデル: 既存の **勤怠修正（correction / Unit 05）** と同じ「申請 → 承認/却下 → 反映」フロー。
> **全 `[Answer]` 確定済み（AI 推奨採用）** → 仕様確定（ゲート①）へ。確定版は [新規機能_有給休暇申請_仕様(SDD).md](./新規機能_有給休暇申請_仕様(SDD).md)。

## お題

社員が**有給休暇を申請**し、**上長が承認/却下**、承認されたら**勤怠に反映**される機能を追加する。
例:「10/20 を有給で休みたい」→ 上長が承認 → その日が有給として記録される。

## ゴール（最低限）

- 社員が有給休暇を申請できる（対象日・理由など）
- 上長が承認待ちの申請を確認し、承認/却下できる
- 承認された有給が勤怠側で分かる（履歴などで確認できる）
- 自分の申請状態（申請中/承認/却下）を確認できる

## 確定方針（重要な前提）

> **本機能は研修用デモ**。「correction と同じ"申請→承認→反映"の型を、スキルにプロセスを任せて短時間で再現する」ことが主眼で、**完成しなくても OK**。
> よって全 Q を **「correction に寄せて最小に倒す」AI 推奨で確定**する。
> ただし各 Q には **「▸ 拡張時」** として、別の選択肢を採る場合の再設計ポイントを最低限記録する（将来ここを起点に再設計できるように）。
> 拡張の全体像は仕様書の「将来拡張マトリクス」に集約する。

## 既存 correction（Unit 05）との対比（設計の土台）

| 観点 | 勤怠修正（既存） | 有給休暇（今回） |
|------|----------------|----------------|
| フロー | 申請 → 上長承認/却下 → 反映 | **同じ型**を踏襲 |
| 承認者 | 申請者と同一部署の上長（`isManager`）。上長は自己承認 | 同じ（Q5） |
| ステータス | PENDING / APPROVED / REJECTED | 同じ（Q13） |
| 楽観ロック | `@Version` + 手動 version チェック（409） | 同じ（Q13） |
| 却下理由 | 必須（`reject` 時に body で受ける） | 同じ（Q6） |
| 承認後の反映 | AttendanceRecord の時刻を更新 or 新規作成し `corrected=true` | **leave テーブルに APPROVED で残す（AttendanceRecord は触らない）（Q9）** |
| 固有概念 | なし | 有給残日数の管理 → **今回は持たない（Q1）** |

## 未決事項（[Question] / [Answer]）→ 全確定

### Q1: 有給残日数の管理をするか？（最重要 / スコープの大分岐）

[Question] 有給の「残日数」を管理しますか？
- (A) **管理しない**（申請を通すだけ。残数チェック・付与・消化計算はしない）
- (B) 残日数を持ち、申請時/承認時に残数をチェック・減算する（付与ロジックも必要）

[Answer] **(A) 管理しない**（AI 推奨採用）。correction と同じ「申請 → 承認 → 反映」の型に集中し、残数管理はスコープ外。
▸ **拡張時 (B)**: `leave_balances` テーブル（社員×年度×付与/消化/残）を新設 or `employees` に残日数カラム追加 → 付与ロジック（勤続年数による付与日数=労基法、入社日起点の付与日計算）→ 年度繰越（2年時効）→ 申請時に残数チェック（不足なら 400/409）→ 承認時に残数を減算（却下/取り下げで戻す）。**この分岐が「デモの型」か「本格的な有給管理」かを決める最大の判断点。**

---

### Q2: 申請の単位は？

[Question] 有給の申請単位はどうしますか？
- (A) **1日単位のみ**（全休）
- (B) 半休（午前/午後）も可
- (C) 時間単位も可

[Answer] **(A) 1日単位（全休）のみ**（AI 推奨採用）。
▸ **拡張時 (B) 半休**: `leave_unit` 区分カラム（FULL/AM/PM）追加 → 残数を 0.5 日単位で扱う（Q1(B) と連動）→ 勤務時間計算（`WorkDuration`）と半休の整合。**拡張時 (C) 時間単位**: 開始/終了時刻カラム + 時間有給の残時間管理。いずれも `WorkDuration` VO との連携が必要。

---

### Q3: 一度の申請で指定できる範囲は？

[Question] 1回の申請で休む日をどう指定しますか？
- (A) **単一日**（1申請=1日）
- (B) 連続した期間（開始日〜終了日）を1申請で
- (C) 飛び石の複数日をまとめて1申請で

[Answer] **(A) 単一日**（AI 推奨採用）。correction の `targetDate`（単一日）と対称で Entity/画面がそのまま倣える。複数日は「申請を複数出す」で代替。
▸ **拡張時 (B) 期間**: `start_date`/`end_date` の2カラム化 → 期間内の営業日数計算（祝日カレンダーはスコープ外のため土日除外のみ）→ Q1(B) と組み合わせると消化日数=営業日数で残数減算。**拡張時 (C) 飛び石**: 申請ヘッダ + 日付明細テーブルに正規化（`leave_request_dates`）。

---

### Q4: 休暇の種別は？

[Question] 扱う休暇の種類は「有給休暇」だけですか？それとも欠勤・特別休暇・慶弔などの種別も選べるようにしますか？

[Answer] **有給休暇のみ**（種別カラムを持たない）（AI 推奨採用）。
▸ **拡張時**: `leave_type` enum カラム（PAID/SPECIAL/CONDOLENCE/ABSENCE…）追加 → 種別ごとの残数管理・集計分岐・画面の種別選択 UI。有給以外は残数管理しない種別もあるため Q1 と分岐が絡む。

---

### Q5: 承認者のルールは correction と同じでよいか？

[Question] 承認者は「申請者と同一部署の上長（`isManager=true`）」、上長本人の申請は自己承認、という correction と同じルールでよいですか？

[Answer] **correction と完全に同じ**（AI 推奨採用）。`validateApprover` パターン再利用（同一部署 && `isManager`）。ロール(ADMIN)特権での承認は入れない。
▸ **拡張時**: 代理承認（上長不在時）→ 部署に副上長/代理者を持たせ `validateApprover` に分岐追加。管理者による強制承認 → ADMIN バイパスを追加。多段承認 → 承認ステップの状態機械化。

---

### Q6: 却下理由は必須か？

[Question] 却下時に理由の入力を必須にしますか？（correction は必須・最大500文字）

[Answer] **必須**（AI 推奨採用）。`reject` 時に body で理由を受け `reject_reason`（最大500文字）に保存。correction と同じ。
▸ **拡張時**: 変更なし（このまま流用でよい）。

---

### Q7: 申請者が自分の申請を取り下げ（キャンセル）できるか？

[Question] 申請者が、承認前（PENDING）の自分の申請を取り下げできるようにしますか？

[Answer] **今回はスコープ外**（AI 推奨採用）。correction にも取り下げ API は無い。申請/承認/却下の3操作で型としては十分。
▸ **拡張時**: `CANCELED` ステータス追加 → `PATCH /api/leave-requests/{id}/cancel`（申請者本人のみ・PENDING のみ・version チェック）→ 承認済みの取り消し（＝残数戻し、Q1(B) と連動）を許すかは別途判断。

---

### Q8: 過去日・当日の申請を許すか？

[Question] 申請できる対象日に制約をつけますか？
- (A) **制約なし**（過去日・当日・未来日いずれも可）
- (B) 未来日（当日含む）のみ。過去日は不可

[Answer] **(A) 制約なし**（AI 推奨採用）。バリデーションを増やさず型に集中。過去日を許しても承認フローは成立する。
▸ **拡張時 (B)**: `targetDate >= 今日` のバリデーションを Backend（Service or Bean Validation のカスタム）と Frontend（`min` 属性 + 送信前チェック）の両方に追加。締切ルール（「N営業日前まで」等）を足すならさらにカレンダー計算が必要。

---

### Q9: 承認された有給を「どこに」反映するか？（correction との最大の違い）

[Question] 承認後、有給が反映される先をどうしますか？
- (A) **有給申請テーブル（leave_requests）に APPROVED 状態で残るだけ**（AttendanceRecord は触らない）
- (B) 承認時に該当日の `attendance_records` に「有給」フラグ付きレコードを作る
- (C) 月次集計（RPT）に有給日数を出す

[Answer] **(A)**（AI 推奨採用）。承認で `status=APPROVED` になり、本人の申請一覧・（あれば）勤怠履歴の「有給申請」表示で確認できる。correction のように既存 AttendanceRecord を書き換えない（時刻の修正ではないため反映先が異なる）。
▸ **拡張時 (B)**: `attendance_records` に `is_leave`/種別カラム追加 or 有給レコード種別を新設 → **既存の「欠勤日数=平日に打刻が無い日」集計ロジックの再定義**（有給日を欠勤から除外）。**拡張時 (C)**: 月次集計（RPT / `MonthlySummary`）に「有給取得日数」列を追加 → CSV/PDF 帳票にも反映。(B)(C) は既存の集計・帳票への波及が大きいので Q1・Q3 の拡張とセットで設計し直す。

---

### Q10: 同じ日の重複をチェックするか？

[Question] 同じ対象日に、既に自分の有給申請 / 打刻レコードがある場合をどう扱いますか？
- (A) **チェックしない**（重複申請も許す）
- (B) 同一日の PENDING/APPROVED が既にあれば弾く（409 など）

[Answer] **(A) チェックしない**（AI 推奨採用）。まず型優先。
▸ **拡張時 (B)**: `create` 冒頭で「同一 requester・同一 targetDate・status ∈ {PENDING, APPROVED}」の存在確認 → あれば 409（correction の二重打刻ガード＝バグ②修正と同じ手法）。打刻レコードとの突合（有給日に打刻がある矛盾チェック）はさらに別途。

---

### Q11: 入力項目と文字数上限は？

[Question] 申請フォームの入力項目は「対象日 + 申請理由」でよいですか？理由の必須可否と文字数上限は？

[Answer] **対象日=必須、理由=必須・最大500文字**（AI 推奨採用）。correction の reason と対称。
▸ **拡張時**: 種別（Q4）・単位（Q2）・期間（Q3）を足すと入力項目が増える。理由を任意にするなら承認側 UI で「理由なし」表示の考慮が要る。

---

### Q12: 閲覧範囲は？

[Question] 有給申請を閲覧できるのは誰ですか？

[Answer] **correction と同じ**（AI 推奨採用）。本人=自分の申請一覧、上長=自部署の承認待ち一覧。管理者(ADMIN)向けの全社一覧は今回スコープ外。
▸ **拡張時**: 管理者向け全社一覧 → `GET /api/leave-requests/all`（`hasRole("ADMIN")`、attendance の `/all` と同じ手法）→ 画面も管理者メニューに追加。

---

### Q13: 楽観ロックは correction と同じ方式でよいか？

[Question] 承認/却下の競合は correction と同じく `@Version` + version チェック（不一致で 409）でよいですか？

[Answer] **correction と完全に同じ**（AI 推奨採用）。承認/却下時にクライアントの version を渡し、不一致なら 409。ステータスは PENDING/APPROVED/REJECTED。
▸ **拡張時**: 変更なし（このまま流用でよい）。CANCELED（Q7）を足す場合はステータス遷移の妥当性チェックを追加。
