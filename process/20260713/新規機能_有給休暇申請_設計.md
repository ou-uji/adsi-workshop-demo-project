# 有給休暇申請機能 — 実装設計（ゲート②）

> 仕様: [新規機能_有給休暇申請_仕様(SDD).md](./新規機能_有給休暇申請_仕様(SDD).md)
> ブランチ: `feature/skill-auto-dev`
> 方針: 既存 **勤怠修正（correction / Unit 05）** の構成を**そのままミラー**し、有給に不要な要素（`attendance_record` 依存・時刻2カラム）を削ぎ落とす。correction のコードを実読して差分を最小化した。

## 設計の基本方針

- **correction をテンプレートにする**。`com.example.attendance.correction` → `com.example.attendance.leave` に写し、フィールドを `targetDate` + `reason` に絞る。
- correction との**唯一の本質的差分** = 承認時の副作用。correction は AttendanceRecord を更新/新規作成するが、**leave は `status=APPROVED` にするだけ**（Q9=A、`attendance_records` は触らない）。→ `approve` が最もシンプルになる。
- 命名: correction の `AttendanceCorrection`/`CorrectionStatus` に対し、`LeaveRequest`/`LeaveStatus`。API パスは `/api/leave-requests`。

## 1. ドメインモデル（ライト DDD）

### Entity: `LeaveRequest`

correction の `AttendanceCorrection` から `attendanceRecord`・`correctedClockIn`・`correctedClockOut` を除去。

| フィールド | 型 | JPA | 備考 |
|-----------|----|----|------|
| `id` | `UUID` | `@Id` | アプリ採番（`UuidCreator.getTimeOrderedEpoch()` = UUIDv7） |
| `requester` | `Employee` | `@ManyToOne(LAZY) @JoinColumn(name="requester_id", nullable=false)` | 申請者 |
| `approver` | `Employee` | `@ManyToOne(LAZY) @JoinColumn(name="approver_id")` | nullable。承認/却下時に設定 |
| `targetDate` | `LocalDate` | `@Column(nullable=false)` | 有給取得日（単一日） |
| `reason` | `String` | `@Column(nullable=false, length=500)` | 申請理由 |
| `status` | `LeaveStatus` | `@Enumerated(STRING) @Column(nullable=false, length=20)` | 初期 PENDING |
| `rejectReason` | `String` | `@Column(length=500)` | nullable。却下時のみ |
| `version` | `Long` | `@Version` | 楽観ロック |
| `createdAt` | `Instant` | `@CreatedDate @Column(nullable=false, updatable=false)` | 監査 |
| `updatedAt` | `Instant` | `@LastModifiedDate @Column(nullable=false)` | 監査 |

クラスアノテーション（correction と同一）: `@Entity @Table(name="leave_requests") @EntityListeners(AuditingEntityListener.class) @Data @NoArgsConstructor @AllArgsConstructor @Builder`

### Enum: `LeaveStatus`

```java
public enum LeaveStatus { PENDING, APPROVED, REJECTED }
```

### Value Object

- **なし**。有給には時刻・勤務時間のような自然な VO が無い（残日数管理もしないため `LeaveBalance` 等も不要）。correction も VO を持たない。

### Repository: `LeaveRequestRepository`

correction のクエリメソッド3本をそのまま踏襲（メソッド名は Spring Data の派生クエリ規約）:

```java
public interface LeaveRequestRepository extends JpaRepository<LeaveRequest, UUID> {
    List<LeaveRequest> findByRequesterIdOrderByCreatedAtDesc(UUID requesterId);
    List<LeaveRequest> findByRequesterIdAndStatusOrderByCreatedAtDesc(UUID requesterId, LeaveStatus status);
    List<LeaveRequest> findByRequesterDepartmentIdAndStatusOrderByCreatedAtDesc(UUID departmentId, LeaveStatus status);
}
```

### Service: `LeaveRequestService`（interface + impl）

correction の5メソッドから、approve の副作用だけ差し替える。

```java
public interface LeaveRequestService {
    LeaveRequestResponse create(UUID requesterId, LeaveRequestCreateRequest request);
    List<LeaveRequestResponse> findByRequester(UUID requesterId, LeaveStatus status);
    List<PendingLeaveRequestResponse> findPending(UUID managerId);
    LeaveRequestResponse approve(UUID leaveRequestId, UUID approverId, Long version);
    LeaveRequestResponse reject(UUID leaveRequestId, UUID approverId, String reason, Long version);
}
```

`LeaveRequestServiceImpl`（`@Slf4j @Service @Transactional(readOnly=true)`、DI は `LeaveRequestRepository` + `EmployeeRepository` の**2つのみ**。correction と違い `AttendanceRecordRepository` は不要）:

- **create**（`@Transactional`）: requester を取得 → `LeaveRequest` を UUIDv7・`status=PENDING` で build → save → `LeaveRequestResponse.from(saved)`
- **findByRequester**: status 有無で派生クエリを分岐（correction と同じ）
- **findPending**: manager の部署 ID を取り、`findByRequesterDepartmentIdAndStatusOrderByCreatedAtDesc(deptId, PENDING)`
- **approve**（`@Transactional`）: `findLeaveRequestOrThrow` → `findEmployeeOrThrow` → `validateApprover` → `validateVersion` → `status=APPROVED`・`approver` セット → save。**AttendanceRecord には一切触れない**（ここが correction との最大の差）
- **reject**（`@Transactional`）: 同上 → `status=REJECTED`・`approver`・`rejectReason` セット → save
- private `validateApprover`（同一部署 && `isManager`、不一致 403）/ `validateVersion`（version 不一致 409）/ `findLeaveRequestOrThrow`（404）/ `findEmployeeOrThrow`（404）— **correction からロジックそのまま流用**

> 補足: correction 同様「既に APPROVED/REJECTED の申請への再承認」を明示ガードしない。`@Version` の bump により古い version での再操作は `validateVersion` で 409 になる（十分）。

## 2. API 設計（correction 準拠 / `/api/leave-requests`）

`@RestController @RequestMapping("/api/leave-requests")`。アイデンティティは `@RequestParam UUID`（correction と同じ既存様式。※セッション非照合は既知事項でアーキ踏襲）。

| メソッド | パス | 引数 | 返却 | ステータス |
|---------|------|------|------|-----------|
| POST | `/api/leave-requests` | `@RequestParam UUID requesterId` + `@Valid @RequestBody LeaveRequestCreateRequest` | `LeaveRequestResponse` | **201** (`@ResponseStatus(CREATED)`) |
| GET | `/api/leave-requests` | `@RequestParam UUID requesterId`, `@RequestParam(required=false) LeaveStatus status` | `List<LeaveRequestResponse>` | 200 |
| GET | `/api/leave-requests/pending` | `@RequestParam UUID managerId` | `List<PendingLeaveRequestResponse>` | 200 |
| PATCH | `/api/leave-requests/{id}/approve` | `@PathVariable UUID id`, `@RequestParam UUID approverId`, `@RequestParam Long version` | `LeaveRequestResponse` | 200 |
| PATCH | `/api/leave-requests/{id}/reject` | `@PathVariable UUID id`, `@RequestParam UUID approverId`, `@Valid @RequestBody LeaveRejectRequest` | `LeaveRequestResponse` | 200 |

### DTO（すべて `record`、correction と対称）

**`LeaveRequestCreateRequest`**（correction の時刻2フィールドを削除）:
```java
public record LeaveRequestCreateRequest(
    @NotNull LocalDate targetDate,
    @NotNull @Size(min = 1, max = 500) String reason
) {}
```

**`LeaveRejectRequest`**（correction の `CorrectionRejectRequest` と同一構造）:
```java
public record LeaveRejectRequest(
    @NotNull @Size(min = 1, max = 500) String reason,
    @NotNull Long version
) {}
```

**`LeaveRequestResponse`**（correction の Response から時刻・attendanceRecordId を削除）: 静的ファクトリ `from(LeaveRequest)` 付き。
```
id, requesterId, requesterName, approverId, approverName,
targetDate, reason, status, rejectReason, version, createdAt
```
（approver は null ガードして id/name を平坦化）

**`PendingLeaveRequestResponse`**（承認待ち用の絞り込み projection）: 静的ファクトリ `from(...)` 付き。
```
id, requesterId, requesterName, targetDate, reason, version, createdAt
```

### エラー→HTTP（既存 `GlobalExceptionHandler` がそのまま処理）

- `EntityNotFoundException` → 404
- `@Valid` 違反（`MethodArgumentNotValidException`）→ 400（フィールド別 errors）
- `ResponseStatusException(FORBIDDEN)` → 403（`validateApprover`）
- `ResponseStatusException(CONFLICT)` / `ObjectOptimisticLockingFailureException` → 409（`validateVersion` + `@Version`）

## 3. DB 設計（Flyway `V6__create_leave_requests.sql`）

correction の V4 から `attendance_record_id`・`corrected_clock_in`・`corrected_clock_out` を除去し、`target_date` + `reason` に。

```sql
CREATE TABLE leave_requests (
    id UUID PRIMARY KEY,
    requester_id UUID NOT NULL REFERENCES employees(id),
    approver_id UUID REFERENCES employees(id),
    target_date DATE NOT NULL,
    reason VARCHAR(500) NOT NULL,
    status VARCHAR(20) NOT NULL CHECK (status IN ('PENDING', 'APPROVED', 'REJECTED')),
    reject_reason VARCHAR(500),
    version BIGINT NOT NULL DEFAULT 0,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL,
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL
);

CREATE INDEX idx_leave_requests_requester ON leave_requests(requester_id);
CREATE INDEX idx_leave_requests_status ON leave_requests(status);
```

- UUID PK（DB デフォルト無し・アプリ採番）、`version BIGINT NOT NULL DEFAULT 0`、監査2カラム NOT NULL — correction と同じ規約。
- 配置: `packages/backend/src/main/resources/db/migration/V6__create_leave_requests.sql`
- **`ddl-auto` 禁止**のため必須。H2（workshop）/ PostgreSQL 両対応の型（`UUID` / `TIMESTAMP WITH TIME ZONE` / `VARCHAR` / `BIGINT`）で書く。

### ER 図（関連）

```
employees 1 ──< leave_requests (requester_id)  必須
employees 1 ──< leave_requests (approver_id)   任意（承認/却下時）
```
（correction と違い attendance_records との関連は無い）

### seed データ

- 今回は**追加しない**（既存 seed で作られる社員・部署・上長を使い、実 API 検証時に PENDING を新規作成する）。correction も専用 seed は持たない。

## 4. SecurityConfig への追記

`configureAuthorization` の correction ブロックの直後に、leave ルートを列挙する（末尾 `denyAll()` の**前**）。correction と同じく全て `authenticated()`（承認権限は Service の `validateApprover` で担保）:

```java
.requestMatchers(HttpMethod.POST, "/api/leave-requests").authenticated()
.requestMatchers(HttpMethod.GET,  "/api/leave-requests").authenticated()
.requestMatchers("/api/leave-requests/pending/**").authenticated()
.requestMatchers("/api/leave-requests/*/approve").authenticated()
.requestMatchers("/api/leave-requests/*/reject").authenticated()
```

> ⚠ `denyAll()` があるため、**列挙漏れは即 403**。実装時に必ず追加する（correction の直後が自然な位置）。

### API クライアント関数（correction と対称。identity/version は query param）

- `createLeaveRequest(requesterId, request)` → `POST /api/leave-requests?requesterId=...`
- `fetchLeaveRequests(requesterId, status?)` → `GET /api/leave-requests?requesterId=...&status=...`
- `fetchPendingLeaveRequests(managerId)` → `GET /api/leave-requests/pending?managerId=...`
- `approveLeaveRequest(id, approverId, version)` → `PATCH /api/leave-requests/{id}/approve?approverId=...&version=...`
- `rejectLeaveRequest(id, approverId, reason, version)` → `PATCH /api/leave-requests/{id}/reject?approverId=...` body `{ reason, version }`

### Hooks（`useLeaveRequests.ts`）

- クエリキー: `LEAVE_KEY = ["leave-requests"]`, `PENDING_KEY = ["leave-requests","pending"]`
- `useLeaveRequests(status?)` — `useQuery`、key `["leave-requests", requesterId, status]`、`enabled: !!requesterId`
- `useCreateLeaveRequest()` — `useMutation`、成功で `LEAVE_KEY` invalidate + `toast.success("有給申請を送信しました")`、失敗で `toast.error(...)`
- `usePendingLeaveRequests()` — `useQuery`、key `["leave-requests","pending", user?.id]`、`enabled: !!user?.isManager`
- `useApproveLeaveRequest()` — `{ id, version }`、両キー invalidate、toast「有給申請を承認しました」、`onError` toast
- `useRejectLeaveRequest()` — `{ id, reason, version }`、両キー invalidate、toast「有給申請を却下しました」、`onError` toast

### コンポーネント（correction をミラー）

- **`LeaveRequestForm`**: `useState` で `targetDate`（`<Input type="date">`）と `reason`（`<textarea maxLength={500}>`）。`isValid = !!targetDate && reason.trim().length > 0`。送信で `createLeaveRequest` → 成功で `/leave` へ `router.push`。送信ボタンは `!isValid || isPending` で disabled、ラベル「送信中... / 申請する」。キャンセルボタンあり。
- **`LeaveRequestList`**: `DataTable<LeaveRequestResponse>`。列: 対象日 / 理由 / ステータス(StatusBadge) / 却下理由。ステータスフィルタ Select（すべて/申請中/承認済/却下、ALL→undefined）。ヘッダに「新規申請」ボタン（`render={<Link href="/leave/new"/>}`）。ローディングは Skeleton。
- **`PendingLeaveRequestList`**: `DataTable<PendingLeaveRequestResponse>`。列: 申請者 / 対象日 / 理由 / アクション（`<LeaveApprovalActions leaveRequestId version/>`）。空メッセージ「承認待ちの有給申請はありません」。
- **`LeaveApprovalActions`**: props `{ leaveRequestId, version }`。承認ボタン → `approveMutation.mutate({id, version})`。却下は `Dialog` で理由 textarea（required, maxLength 500）→ `rejectMutation.mutate({id, reason, version})`。pending 中 disabled。

### StatusBadge 設定（correction と同一ラベル）

```
PENDING  → { label: "申請中", variant: "secondary" }
APPROVED → { label: "承認済", variant: "default" }
REJECTED → { label: "却下",   variant: "destructive" }
```

### Sidebar 追記（`src/components/layout/Sidebar.tsx`）

- `commonItems` に `{ title: "有給申請", href: "/leave", icon: <lucide の適切なアイコン、例: CalendarDays> }`
- `managerItems` に `{ title: "有給承認", href: "/leave-approvals", icon: CalendarCheck }`（`user?.isManager` のときのみ表示）

### 日付の扱い

- correction は時刻を `+09:00` オフセットで ISO 化していたが、**leave は `targetDate` が `LocalDate`（日付のみ）**なので、`<input type="date">` の値（`"YYYY-MM-DD"`）をそのまま送る。タイムゾーン変換は不要。

## 9. correction からの差分サマリー（レビュー観点）

| 項目 | correction | leave（今回） |
|------|-----------|--------------|
| Entity フィールド | attendanceRecord + 時刻2つ + targetDate + reason | **targetDate + reason のみ**（attendanceRecord・時刻を削除） |
| Service DI | 3 repo（correction/attendanceRecord/employee） | **2 repo**（leaveRequest/employee）。AttendanceRecordRepository 不要 |
| approve の副作用 | AttendanceRecord を更新 or 新規作成し corrected=true | **status=APPROVED にするだけ**（副作用なし） |
| CreateRequest | attendanceRecordId? + targetDate + 時刻2つ + reason | **targetDate + reason のみ** |
| Response | + attendanceRecordId, correctedClockIn/Out | **それらを削除** |
| Frontend Form | date + time×2 + reason、ISO 変換あり | **date + reason のみ**、ISO 変換なし |
| 承認画面 | `/approvals`（既存） | **`/leave-approvals`（新規・別ルート）** |

→ 差分は「削る」方向が中心で、リスクは低い。承認の副作用が消えるため approve は correction より単純。
