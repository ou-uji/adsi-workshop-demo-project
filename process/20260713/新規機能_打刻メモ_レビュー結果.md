## 却下した指摘

- test-reviewer が付けた「CRITICAL（連続編集で state 残留）」は、統合テスト（Backend）の話に Frontend の `setMemo` を混同した**ファイル取り違え**。実装上 `MemoEditDialog` は `record` を `useEffect` で毎回同期しており該当しない → 却下。

## known-issue（別途・全体対応の候補）

**認可のセッション非照合**: attendance / correction の全 API が `employeeId`/`managerId` 等をクライアントから受け取り、`SecurityContextHolder` のセッションユーザと照合していない。他人の ID を指定すると他人のデータを操作できる理論上の余地がある（フロントは自分の ID しか送らないが、API 直叩きで回避可能）。
- 影響: 打刻メモに限らず既存機能全体。
- 対応方針（未実施）: `AuthServiceImpl.getCurrentUser()` と同様に `SecurityContextHolder` からセッションユーザを取得し、各 API で employeeId 一致を検証（または employeeId 引数自体を廃してセッションユーザを使う）。
- 本ワークショップのスコープ外。将来 Issue 化を推奨。
